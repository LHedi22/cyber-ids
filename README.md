# CyberIDS — AI-Augmented Intrusion Detection System

CyberIDS is a cloud-native intrusion detection system that layers three complementary detection strategies — signature-based Snort 3 rules, an unsupervised Isolation Forest anomaly detector, and an LLM explanation engine — into a single real-time SOC dashboard. Where legacy IDS tools produce raw alert floods that overwhelm analysts, CyberIDS correlates, deduplicates, and translates every detection into a plain-English sentence generated by a locally-running Phi-3 mini model, so the analyst sees *what happened*, *why it was flagged*, and *which traffic features drove the score* — all without sending data to an external API.

---

## Architecture

```
[Snort 3 / Log Simulator]
        ↓ writes → /data/snort_alerts.jsonl  (append-only, one JSON object per line)
[Log Parser & Normalizer]        pipeline/parser.py
        ↓
[Feature Extractor]              pipeline/features.py   ← extensibility/plugin layer
     ↓             ↓
[Rule Engine]   [Isolation Forest]
pipeline/        pipeline/
rule_engine.py   ml_detector.py
     ↓             ↓
[Alert Aggregator & Deduplicator]   pipeline/aggregator.py
        ↓
[SHAP Explainer]                    pipeline/explainer.py
        ↓
[LLM Interpreter]                   pipeline/llm.py
(Phi-3 mini via Ollama, <5B params)
        ↓
[FastAPI Backend]    dashboard/api.py
        ↓       ↓
[Web Dashboard]   [alerts.jsonl log]
dashboard/        data/alerts.jsonl
static/index.html
```

## Quick Start

### Docker Compose (3 commands)

```bash
# 1. Copy environment config
cp .env .env.local          # edit OLLAMA_HOST or thresholds if needed

# 2. Build images and start all services
docker compose up --build -d

# 3. Open the dashboard
open http://localhost:8000   # Linux: xdg-open http://localhost:8000
```

The first start downloads Phi-3 mini (~2 GB). Subsequent starts use the model cached in the `ollama_data` volume. If no saved model is found, the pipeline auto-trains the Isolation Forest on startup before accepting alerts.

VaultBank target app is available at `http://localhost:5000`.

### Kubernetes (5 commands, including minikube setup)

```bash
# 1. Start minikube with enough memory for Ollama
minikube start --memory=5500 --cpus=4

# 2. Enable the NGINX ingress controller
minikube addons enable ingress

# 3. Build images directly into minikube's Docker daemon
eval $(minikube docker-env)
docker build -f Dockerfile.simulator  -t cyber-ids-simulator:latest .
docker build -f Dockerfile.pipeline   -t cyber-ids-pipeline:latest  .
docker build -f target-app/Dockerfile -t cyber-ids-target:latest    ./target-app

# 4. Apply all manifests
kubectl apply -f k8s/

# 5. Add the local DNS entry and open the dashboard
echo "$(minikube ip) cyber-ids.local" | sudo tee -a /etc/hosts
open http://cyber-ids.local
```

Watch pod status with `kubectl get pods -n cyber-ids -w`. The `ollama` pod shows `Init:0/1` while Phi-3 mini is being pulled — this is expected and takes 2–5 minutes on the first run. The model is cached in the `ollama-data-pvc` volume and reused on subsequent starts.

## How Detection Works

### Rule-Based Layer (Snort 3 + Rule Engine)

Network traffic is captured by Snort 3 — or, in `DEMO_MODE=true`, by the Python log simulator — and matched against five custom detection signatures covering SSH brute force (SID 1000001), SQL injection in HTTP URIs (SID 1000002), horizontal port scanning (SID 1000003), off-hours authenticated access (SID 1000004), and large outbound data transfers consistent with exfiltration (SID 1000005). Each match is written as a structured JSON line to `/data/snort_alerts.jsonl`. The Python rule engine (`pipeline/rule_engine.py`) then applies a second pass of configurable threshold logic — for example, firing `BRUTE_FORCE` only when `failed_logins_60s` exceeds the operator-tunable threshold — decoupling sensitivity tuning from the underlying signature definitions.

### ML Layer (Isolation Forest)

Every alert triggers a sliding-window feature extraction step (`pipeline/features.py`) that computes eight numeric signals from the last 60 seconds of traffic attributed to the source IP: failed login count, unique ports contacted, requests per minute, off-hours flag, new-IP flag, bytes out, unique destination IPs, and error rate. These features are passed to a pre-trained Isolation Forest (`pipeline/ml_detector.py`) that was trained exclusively on synthetic normal traffic. Any point the model struggles to isolate — because it sits far outside the dense normal cluster — receives a high anomaly score and is flagged as `ML_ANOMALY`. The model ships pre-baked into the Docker image (trained at build time by `train.py`) and auto-retrains on startup if the saved model file is absent, so the system is always operational without manual intervention.

### AI Explanation Layer (SHAP + LLM)

For every flagged alert, SHAP (SHapley Additive exPlanations) decomposes the anomaly score into per-feature contributions (`pipeline/explainer.py`). The top contributing features — labelled, ranked, and quantified — are passed as structured context to Phi-3 mini running locally via Ollama (`pipeline/llm.py`), which returns a single plain-English sentence explaining the threat in analyst-readable terms. If Ollama is unreachable, the pipeline falls back to a deterministic template explanation derived from the SHAP factors and continues processing without interruption. The explanation, SHAP values, and anomaly score are all persisted in the alert record and surfaced in the dashboard's detail panel alongside a proportional bar chart of feature contributions.

## Demo

### Triggering Attacks

**From the dashboard:** Click the **Simulate Attack** button in the header. This calls `POST /api/simulate`, which instructs the simulator to emit a burst of attack traffic across all five scenarios in sequence, with realistic normal traffic interspersed between attacks.

**Via curl:**

```bash
# Trigger one full attack sequence (Docker Compose)
curl -X POST http://localhost:8000/api/simulate

# Trigger on Kubernetes
curl -X POST http://cyber-ids.local/api/simulate
```

**Against VaultBank directly** (for a hands-on demo that shows the target app responding):

```bash
# SQL injection bypass on the login form
curl -X POST http://localhost:5000/login \
  -d "username=' OR '1'='1&password=anything"

# Authorization bypass on the admin panel
curl "http://localhost:5000/admin?override=true"

# Brute-force simulation (requires hydra or manual loop)
for i in $(seq 1 20); do
  curl -s -X POST http://localhost:5000/login \
    -d "username=admin&password=wrong${i}" > /dev/null
done
```

### What to Expect

Within 2–3 seconds of triggering attacks, new rows appear at the top of the alert table with a yellow slide-in flash animation. Each row shows:

| Column | Description |
|---|---|
| **Time** | Timestamp of the originating Snort alert |
| **Severity** | `CRITICAL` (red) · `HIGH` (orange) · `MEDIUM` (yellow) · `LOW` (green) |
| **Type** | `BRUTE_FORCE` · `PORT_SCAN` · `SQL_INJECTION` · `OFF_HOURS` · `DATA_EXFIL` · `ML_ANOMALY` |
| **Source IP** | Click to copy to clipboard |
| **Detection Source** | `Rule` (blue) · `ML` (purple) · `Both` (green) when both layers fire |
| **Explanation** | One-sentence LLM summary of the threat |

Click any row to expand the full explanation panel at the bottom of the page, which includes the SHAP contribution bar chart showing exactly which traffic features drove the detection score. The stats row (Total / Critical / High / Medium / Alerts per minute) updates in real time via WebSocket. If the connection drops, the dashboard reconnects automatically with exponential backoff (1 s → 2 s → 4 s → 8 s → max 30 s) and falls back to polling `GET /api/stats` every 10 seconds.

## Configuration

All configuration is read from environment variables at startup. Edit `.env` to adjust thresholds, paths, or model settings — no code changes required.

| Variable | Default | Description |
|---|---|---|
| `OLLAMA_HOST` | `http://ollama:11434` | Ollama inference server URL |
| `OLLAMA_MODEL` | `phi3:mini` | LLM model name for alert explanations |
| `ALERT_LOG_PATH` | `/data/alerts.jsonl` | Output path for processed alert log |
| `SNORT_ALERT_PATH` | `/data/snort_alerts.jsonl` | Input path for raw Snort or simulator alerts |
| `ISOLATION_FOREST_CONTAMINATION` | `0.05` | Expected anomaly fraction (0.0–0.5); lower reduces false positives |
| `MODEL_PATH` | `models/isolation_forest.pkl` | Path to the saved Isolation Forest model bundle |
| `SLIDING_WINDOW_SECONDS` | `60` | Feature extraction lookback window in seconds |
| `BRUTE_FORCE_THRESHOLD` | `5` | Failed logins per window before `BRUTE_FORCE` fires |
| `PORT_SCAN_THRESHOLD` | `20` | Unique ports per 10 s before `PORT_SCAN` fires |
| `DATA_EXFIL_BYTES` | `10000` | Outbound bytes per window before `DATA_EXFIL` fires |
| `ALERT_DEDUP_SECONDS` | `30` | Suppress duplicate alerts from the same source within this window |
| `DASHBOARD_PORT` | `8000` | FastAPI server port |
| `LOG_LEVEL` | `INFO` | Python logging level (`DEBUG`, `INFO`, `WARNING`, `ERROR`) |
| `DEMO_MODE` | `true` | `true` = Python simulator; `false` = read live Snort output |

To retrain the model after changing thresholds or feature distributions:

```bash
python train.py                          # default: 10,000 normal samples
python train.py --samples 20000 --seed 7 # larger training set
```

## Extending the System

### Adding a new detection rule

1. **Write a Snort rule** in `snort/rules/custom.rules` using the next available SID (e.g., `1000006`). Follow the existing pattern — set `msg`, `flow`, any `content` match, and a `detection_filter` if the rule is rate-based.

2. **Add a rule engine check** in `pipeline/rule_engine.py` inside the `evaluate()` method. Map the new SID or feature threshold to an alert type string and severity level:
   ```python
   if fv.features["your_new_feature"] > YOUR_THRESHOLD:
       yield self._make_alert("YOUR_TYPE", "HIGH", fv)
   ```

3. **Add a simulator scenario** in `simulator/log_generator.py` so the new rule fires in `DEMO_MODE=true`. Follow the existing `_attack_*` generator pattern and register it in the round-robin scheduler so it fires during a `Simulate Attack` call.

4. **Update the feature extractor** in `pipeline/features.py` if the new rule depends on a signal not already in the eight-feature vector. Add the new key to `FEATURE_ORDER` in `pipeline/ml_detector.py` to keep the ML layer consistent.

5. **Retrain the model** to incorporate the new feature distribution:
   ```bash
   python train.py
   docker build -f Dockerfile.pipeline -t cyber-ids-pipeline:latest .
   ```
   The rebuilt image ships the updated model baked in, so no volume mounts or manual copies are needed.

### Adding a new log source

Replace or extend the `_parse_line()` method in `pipeline/parser.py` to accept a different JSON schema. The rest of the pipeline only consumes `LogEvent` dataclasses — as long as your parser emits valid `LogEvent` objects, no other file needs to change. For a completely different transport (syslog, Kafka, cloud logging), wrap your consumer in a generator and feed it into the same `tail()` loop that currently reads the JSONL file.

## Tech Stack

| Component | Technology |
|---|---|
| Packet inspection / signature detection | Snort 3 (open-source IPS) |
| Log simulation (demo mode) | Python 3.11 — `simulator/log_generator.py` |
| Log parsing & feature extraction | Python 3.11 — `pipeline/parser.py`, `pipeline/features.py` |
| Rule-based detection | Python 3.11 — `pipeline/rule_engine.py` |
| Anomaly detection | scikit-learn `IsolationForest` — `pipeline/ml_detector.py` |
| Explainability | SHAP TreeExplainer — `pipeline/explainer.py` |
| LLM interpretation | Phi-3 mini 3.8B via Ollama — `pipeline/llm.py` |
| Alert aggregation & deduplication | Python 3.11 — `pipeline/aggregator.py` |
| API & WebSocket server | FastAPI + Uvicorn — `dashboard/api.py` |
| SOC dashboard | Vanilla JS/CSS, no framework, no CDN — `dashboard/static/index.html` |
| Target vulnerable app | Flask 3 — `target-app/app.py` (VaultBank) |
| Model serialisation | joblib — saves/loads the trained model bundle |
| Container runtime | Docker Compose (development) |
| Container orchestration | Kubernetes / minikube (production demo) |
| Ingress | NGINX Ingress Controller |
